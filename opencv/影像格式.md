🧠 一、影像格式是什麼？

影像格式（Image Format）＝像素排列方式 + 顏色編碼方式 + 記憶體儲存布局

簡單說：

同一張圖的畫面內容一樣，但在 RAM 裡是怎麼排資料的、每個像素佔幾個 byte、顏色通道順序為何，都由格式定義。

🎨 二、顏色編碼（Color Encoding）

主要分成兩大類：

| 類型                 | 說明                          | 範例                                  |
| ------------------ | --------------------------- | ----------------------------------- |
| **RGB 系列**         | 直接儲存紅、綠、藍通道，適合顯示與運算         | RGB24、BGR24、RGBA32、BGRA32           |
| **YUV / YCbCr 系列** | 分離亮度(Y)與色度(U,V)，適合影像壓縮與視訊編碼 | YUV420、YUV422、YUV444、NV12、YU12、YV12 |

🌈 三、RGB 類格式

| 格式         | 每像素位元數            | 排列順序       | 備註                     |
| ---------- | ----------------- | ---------- | ---------------------- |
| **RGB24**  | 24 bits (3 bytes) | R, G, B    | 常見於 OpenCV 的 `CV_8UC3` |
| **BGR24**  | 24 bits (3 bytes) | B, G, R    | OpenCV 的預設色彩空間         |
| **RGBA32** | 32 bits (4 bytes) | R, G, B, A | 含透明度通道                 |
| **BGRA32** | 32 bits (4 bytes) | B, G, R, A | GPU / GStreamer 常用格式   |

🎯 在 OpenCV 中：

cv::Mat(type=CV_8UC3) 通常是 BGR。

CUDA kernel 若是從 NV12 或 YUV 轉換，最常見的輸出是 BGR or RGBA。

📺 四、YUV / YCbCr 類格式

這系列才是視訊系統的主戰場。

1️⃣ 色彩取樣 (Chroma Subsampling)

| 模式        | 意義          | 比例           | 說明                  |
| --------- | ----------- | ------------ | ------------------- |
| **4:4:4** | 無壓縮         | 每像素都有獨立 YUV  | 色彩最準                |
| **4:2:2** | 水平壓一半       | 每兩個像素共用 U,V  | 常見於 HDMI、相機介面       |
| **4:2:0** | 水平 + 垂直各壓一半 | 每 4 像素共用 U,V | 常見於 H.264、NV12、YU12 |

YUV420 各種排法

假設一張 4x4 圖：

``` bash
🟢 YU12 (I420)
Y plane: 16 bytes  (4x4)
U plane: 4 bytes   (2x2)
V plane: 4 bytes   (2x2)
排列: [Y全][U全][V全]
```

記憶體順序：

``` bash
YYYYYYYYYYYYYYYY UU UU VV VV
```

🔵 YV12

與 YU12 幾乎相同，只是 U 與 V 交換：

``` bash
排列: [Y全][V全][U全]
```

👉 有些舊版顯示驅動或 DirectShow filter 用 YV12。

🟣 NV12

``` bash
Y plane: 16 bytes
UV plane: 8 bytes (interleaved)
排列: [Y全][UVUVUVUV]
```

即每個 UV 為一對交錯（U0,V0,U1,V1,…）
👉 NVIDIA、Intel GPU、V4L2、GStreamer 都喜歡這種。

🔸 NV21

與 NV12 類似，但順序反過來：

``` bash
[Y全][VUVUVU...]
```

👉 常見於 Android Camera。

⚙️ 五、記憶體對齊（Pitch / Stride）

實務上 GPU、V4L2、GStreamer 傳出影像時，可能不是連續的。

pitch = 每行實際記憶體步距

通常 ≥ width，對齊到 32 / 64 / 128 bytes

因此處理 YUV buffer 時：

``` cpp
uint8_t* y_plane = base + 0 * pitch * height;
uint8_t* u_plane = base + pitch * height;
uint8_t* v_plane = base + pitch * height + (pitch/2 * height/2);
```

不能單純用 width * height 算偏移！

🧩 六、格式轉換常見路徑

| 來源         | 目標                                                       | 常用函式 |
| ---------- | -------------------------------------------------------- | ---- |
| NV12 → BGR | `cv::cuda::cvtColor(GpuMat, GpuMat, COLOR_YUV2BGR_NV12)` |      |
| YU12 → BGR | `cv::cvtColor(Mat, Mat, COLOR_YUV2BGR_I420)`             |      |
| RGBA → BGR | `cv::cuda::cvtColor(GpuMat, GpuMat, COLOR_RGBA2BGR)`     |      |
| BGR → NV12 | 通常需拆出 Y plane + interleaved UV 自行組合                      |      |

🧠 七、實務建議

1. 判斷來源格式是關鍵
    - RTSP / camera / decoder 都會輸出特定格式（NV12 最常見）。
2. GPU pipeline 儘量避免轉換多次
    - 每次 color convert 都是一次 full copy。
3. 若需自己 wrap buffer（如 NvBufSurface）
    - 先看 surface->colorFormat
    - 再依格式判斷 plane 數量與 pitch
    - 再用 cv::cuda::GpuMat 包起來。

🧮 八、示意圖簡表（以 4x4 為例）

| 格式              | Y 平面             | U/V 平面      |
| --------------- | ---------------- | ----------- |
| **YU12 (I420)** | YYYYYYYYYYYYYYYY | UUUU + VVVV |
| **YV12**        | YYYYYYYYYYYYYYYY | VVVV + UUUU |
| **NV12**        | YYYYYYYYYYYYYYYY | UVUVUVUV    |
| **NV21**        | YYYYYYYYYYYYYYYY | VUVUVUVU    |

### RGB YUV / YCbCr 互換損耗

## 🧠 一、RGB vs YUV / YCbCr 的本質差異

|屬性|RGB|YUV / YCbCr|
|---|---|---|
|空間定義|加法混色 (R,G,B)|亮度 + 色度 (Y + U/V)|
|人眼模型|線性光強|模擬人眼感知亮度與顏色|
|適用場景|顯示、渲染、GPU運算|視訊壓縮、傳輸、儲存|
|精度特性|三通道均等|色度常被降採樣 (4:2:0)|

👉 簡單說，YUV 是「壓縮友善」版本的 RGB。  
Y 代表亮度 (Luma)，U/V 代表顏色偏移 (Chroma)。

---

## 🎨 二、理論轉換公式（8-bit範例）

假設輸入是 **sRGB [0–255]**，使用 Rec.601 標準（SDTV 常用）：

### RGB → YCbCr

``` bash
Y  =  0.299R + 0.587G + 0.114B
Cb = -0.1687R - 0.3313G + 0.5B + 128
Cr =  0.5R - 0.4187G - 0.0813B + 128
```

### YCbCr → RGB

```bash
R = Y + 1.402 (Cr - 128)
G = Y - 0.344 (Cb - 128) - 0.714 (Cr - 128)
B = Y + 1.772 (Cb - 128)```

🔸 Rec.709（HDTV）、Rec.2020（UHDTV）係數略有不同，差異約 ±1~2%。

## ⚠️ 三、損耗來源總表

|類型|主要原因|結果|可否避免|
|---|---|---|---|
|**量化誤差**|係數小數捨入、8bit 範圍|顏色偏移 ±1~2 灰階|幾乎不可避免|
|**取樣損失 (chroma subsampling)**|U/V 空間壓縮 (如 4:2:0)|顏色邊緣模糊、色帶|🚫 無法逆轉|
|**gamma / 色域不匹配**|sRGB ↔ BT.709 ↔ BT.601 差異|飽和度或亮度失真|✅ 可校正|
|**矩陣精度誤差**|GPU shader / CUDA kernel 精度不足|色偏或 banding|✅ 用 float 修正|
|**多次轉換**|RGB→YUV→RGB 連續多次|累積誤差、色彩漂移|🚫 降低次數|
## 🧩 四、最常見的損耗實例

### 🔹 1. NV12 → BGR → NV12

每次轉換都：

- 捨入 YUV 浮點小數 → 8bit 整數
    
- 捨入 RGB 浮點 → 整數
    

> 結果：邊界與漸層會產生「顏色顫動」（color jitter）

### 🔹 2. 4:2:0 色度取樣

``` bash
Y plane: 1920x1080
U plane: 960x540
V plane: 960x540
```

當你回轉成 RGB 時，U/V 需要「上採樣」，  
最常用 bilinear → 導致色邊模糊。

## 📊 五、實際數據觀察（ΔE 色差）

| 測試條件                  | ΔE 平均誤差 | 主觀效果     |
| --------------------- | ------- | -------- |
| RGB ↔ YCbCr(444)      | ≈ 0.3   | 肉眼幾乎無差   |
| RGB ↔ YCbCr(420)      | ≈ 1.5–3 | 細節區略模糊   |
| RGB ↔ YCbCr(420) 多次循環 | ≈ 5+    | 顏色開始可見偏移 |

ΔE > 2 就可能被肉眼察覺。

## 🧠 六、工程實務避損策略

1. **盡量在同一色域中運算**  
    → 在 GPU pipeline 中固定格式（例如全程用 NV12 或 RGBA）。
2. **避免重複轉換**  
    → 若需要 BGR 處理，只轉換一次。
3. **使用高 bit depth（10-bit / 16-bit YUV）**  
    → 降低量化誤差。
4. **選擇正確矩陣（Rec.709 vs Rec.601）**  
    → 例如 GStreamer 中設定 `colorimetry=bt709`。
5. **對比色校正（color calibration）**  
    → 若是攝影影像，必須在 RGB 空間做白平衡。

## 💡 七、實務建議（for GPU / GStreamer）

|場景|建議格式|說明|
|---|---|---|
|CUDA 前處理|NV12 (YUV420)|雙平面、零拷貝方便|
|深度學習前處理|RGBA / BGRx|與模型對齊|
|多階段影像串流|固定 NV12 全流程|避免反覆轉換|
|高畫質後製|YUV444 or FP16 RGB|保留色彩精度|

## 🧩 八、關鍵結論

| 問題             | 答案                      |
| -------------- | ----------------------- |
| RGB ↔ YUV 有損嗎？ | ✅ 有損。即使是 4:4:4 也有量化誤差。  |
| 哪種最嚴重？         | 4:2:0 下的多次轉換。           |
| 怎麼避免？          | 減少轉換次數、使用高位元深度與一致的色域矩陣。 |
| 為何影像會偏色？       | 通常是 Rec.601/709 不匹配造成。  |

## Rec.601/709 不匹配

影像轉換時「畫面忽然變灰、顏色怪怪的」的真兇。  
我們來拆解 **Rec.601 / Rec.709 不匹配** 問題──這其實是一個**色域矩陣不一致（color primaries + transfer matrix）**的坑。

## 🎯 一、Rec.601 vs Rec.709 是什麼？

這兩個是 **ITU-R 的顏色標準（colorimetry standards）**，  
用來定義「亮度與色度怎麼從 RGB 映射到 YCbCr」。

|標準|主要應用|解析度範圍|年代|
|---|---|---|---|
|**Rec.601 (BT.601)**|SDTV (480i/p, 576i/p)|≤ 720x576|1982|
|**Rec.709 (BT.709)**|HDTV (720p, 1080p)|≥ 720p|1990|
|**Rec.2020 (BT.2020)**|UHDTV (4K/8K)|≥ 2160p|2012|

## 🧮 二、轉換矩陣差異（8-bit 範例）

> 🎨 把這個想成「RGB ↔ YCbCr 的線性方程式」，不同標準 → 不同係數。

### 🔹 Rec.601 (SD)

```bash
Y  =  0.299R + 0.587G + 0.114B
Cb = -0.1687R - 0.3313G + 0.500B + 128
Cr =  0.500R - 0.4187G - 0.0813B + 128
```

### 🔹 Rec.709 (HD)

``` bash
Y  =  0.2126R + 0.7152G + 0.0722B
Cb = -0.1146R - 0.3854G + 0.500B + 128
Cr =  0.500R - 0.4542G - 0.0458B + 128
```

> 注意：Rec.709 對綠色權重更重，紅藍稍弱。  
> 所以如果用錯矩陣，畫面會**偏紅或偏綠、亮度對比變怪**。

---

## 🧩 三、為什麼會不匹配？

因為各系統預設不同：

|系統|預設 colorimetry|備註|
|---|---|---|
|V4L2 / GStreamer|依據解析度自動判斷 (SD=601, HD=709)|但有時 metadata 缺失！|
|NVIDIA NVDEC / Jetson|常預設 Rec.709|即使是低解析度影片也會|
|OpenCV|不處理色域，直接用固定矩陣|所以你要自己指定|
|ffmpeg|根據流 metadata (`color_primaries`, `matrix_coeffs`)|但許多攝影機沒正確標出|

👉 結果就是：

> 影片解出來是 NV12（Rec.709），但你用 Rec.601 轉 BGR，畫面變「偏紅+發灰」。

---

## 📸 四、實際視覺差異示意

|狀況|現象|
|---|---|
|用 Rec.601 decode HD 影片|整體偏紅、飽和度下降|
|用 Rec.709 decode SD 影片|顏色偏藍綠、亮度變高|
|Rec.709 RGB 在 Rec.601 螢幕顯示|明暗對比錯誤、膚色怪|

## ⚙️ 五、OpenCV / GStreamer / FFmpeg 實務對應

### 🧠 OpenCV

``` cpp
// 預設用 Rec.601
cv::cvtColor(src, dst, cv::COLOR_YUV2BGR_NV12);

// 若你知道是 Rec.709，可用
cv::cvtColor(src, dst, cv::COLOR_YUV2BGR_NV12_709);
```

### 🧩 GStreamer

可明確指定：

``` bash
video/x-raw, format=NV12, colorimetry=bt709, interlace-mode=progressive
```

或在命令列：

```bash
gst-launch-1.0 filesrc location=video.mp4 ! decodebin ! \
'video/x-raw(colorimetry=bt709)' ! videoconvert ! autovideosink
```

### 🔧 FFmpeg

查看 metadata：

``` bash
ffprobe -show_streams video.mp4 | grep color
```

強制指定矩陣：

``` bash 
ffmpeg -i input.mp4 -color_primaries bt709 -color_trc bt709 -colorspace bt709 output.mp4
```

---

## 🧠 六、如何在程式中避免災難

1. **永遠明確指定 colorimetry**
    - 不要讓 GStreamer / decoder 自動猜。
2. **解析 metadata → 選對矩陣**
    - ffprobe、gst-discoverer 都能取。
3. **避免多次轉換（尤其不同標準混用）**
    - 每次轉換都會有亮度再量化誤差。
4. **用 float pipeline / 16bit precision**
    - 減少矩陣換算的捨入誤差。
5. **自訂轉換矩陣**
    - 若你確定來源標準，可自行套用 CUDA kernel。

## 🔚 八、結論

|問題|答案|
|---|---|
|為什麼會變色？|因為用錯了 Rec.601 / Rec.709 轉換矩陣。|
|哪種更常見？|HD (≥720p) 幾乎都是 Rec.709。|
|怎麼避免？|明確指定 colorimetry、不要依預設。|
|可否修復？|若知道原矩陣，可反轉計算修正回來。|
